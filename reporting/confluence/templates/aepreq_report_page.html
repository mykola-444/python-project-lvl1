Glossary

ID Term Description
1 Use Case Defining how program end users will use implemented product requirements on their side. It should contain clear actions or steps to identify the requirement achieved.
2 Positive Scenario Positive Scenario testing determines that your application works as expected. If an error is encountered during Positive Scenario testing, the test fails and new bug should be created in JIRA.
3 Negative Scenario Negative Scenario testing ensures that your application can gracefully handle invalid input or unexpected user behavior. For example, if a user tries to type a letter in a numeric field, the correct behavior in this case would be to display the “Incorrect data type, please enter a number” message. The purpose of negative testing is to detect such situations and prevent applications from crashing. Also, negative testing helps you improve the quality of your application and find its weak points. If Negative scenario fails it triggers a bug in JIRA.
4 Boundary Cases Boundary Case is a behavior of a system when one of its inputs is at or just beyond its maximum or minimum limits.
5 Alternative Flow An Alternate Flow is a step or a sequence of steps that achieves the use case’s goal following different steps than described in the general scenario.
6 GUI Checks GUI testing is the process of ensuring proper functionality of the graphical user interface (GUI), making sure it conforms to its specifications evaluating design elements such as layout, colors, fonts, font sizes, labels, text boxes, text formatting, captions, buttons, lists, icons, links and content, language localisation.
7 Label in JIRA Label used in JIRA Zephyr to identify all tests developed for the related feature. It could be used simply to get all tests covering a product requirement and also report when tests integrated with CI.
8 Test in gerrit Test developed and committed into test cases source.
